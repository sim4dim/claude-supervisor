Coordinator Types

Use the Coordinator panel to dispatch work to agents in other project sessions. Choose the type that matches your intent.

Type What happens When to use
research Agent investigates and reports findings — read-only, no changes Understand a codebase, look up behavior, gather facts
review Code or approach review with structured feedback Get a second opinion before merging or deploying
action Agent makes changes in the target project Delegate implementation work to another session
plan Triggers Moltke pattern — planner proposes, adversarial reviewer finds blockers Architecture decisions, API design, migration strategy
feasibility Same as plan — adversarial check before commitment Hardware purchases, third-party integrations, irreversible changes
debate Three-agent structured debate: PRO agent vs CON agent, active moderator Unclear tradeoffs, technology choices, conflicting requirements
How plan / feasibility works (Moltke pattern)

Named after the Prussian field marshal who stress-tested every plan before committing.

  1. A Planner agent proposes an approach, enumerates assumptions, and identifies risks.
  2. A Moltke (adversarial) agent reads the plan and actively tries to break it — finding hidden costs, dependency conflicts, integration gotchas, and timeline risks.
  3. Both outputs appear in the Operations panel. You decide whether to proceed.
Tip: Run feasibility before any hardware purchase or major dependency addition. It costs one agent turn and can save days of rework.
How debate works

A structured 5-round protocol using a shared chat room (debate-{id}):

  1. Persona setup — AI proposes expert personas for each side. Human approves before proceeding. Skip with --auto.
  2. Round 1: Opening arguments — PRO and CON agents argue in parallel, each building their full case.
  3. Round 2: Rebuttals — Both sides read each other's opening and respond directly, in parallel.
  4. Round 3: Moderator challenge + defense — Moderator asks hard questions; both sides respond.
  5. Round 4: Gap finder — Independent agent reviews the full exchange and surfaces blind spots neither side addressed.
  6. Round 5: Final statements + verdict — Each side closes; moderator issues a balanced synthesis with a recommendation.

Flags: --debaters N (number of positions, default 2), --auto (skip persona approval), --project <name> (run in another project's session).

Tip: Use /debate when you've been going back and forth on a decision. The gap-finder in Round 4 regularly surfaces considerations neither side thought of.

Agent Types

Researcher

Read-only investigation. Reads files, searches code, reports findings. Never modifies anything.

Implementer

Makes code changes, runs tests, fixes bugs. Full read-write access within the project.

Planner

Proposes approaches and architecture. Output is a structured plan document, not code.

Reviewer / Moltke

Adversarial review. Reads a plan or diff and actively tries to find failure modes before you commit.

All agents have background: true so their output doesn't flood your terminal. Results appear in the Agent Activity panel and via MQTT messages.

Chat Rooms

Chat rooms let multiple agents share findings without polling. Messages are retained on individual MQTT topics so none are lost.

sv chat commands

sv chat post   <room> "message"    # post, returns seq number
sv chat history <room>            # read all messages in order
sv chat wait   <room> <seq>       # block until message N arrives
sv chat read   <room>            # read latest message
sv chat clear  <room>            # remove all messages
sv chat init   <room>            # initialize (optional)

Auto-created rooms

Room name Created by Purpose
coordinator-{id} Coordinator dispatch Shared context between dispatched agents in a multi-step request
debate-{id} /debate or debate coordinator type PRO, CON, and moderator exchange during a structured debate
Example: two agents sharing findings 
# Agent A: researcher publishes findings
export SV_CHAT_ROOM="design-review"
sv chat history "$SV_CHAT_ROOM"       # read what others posted first
sv chat post   "$SV_CHAT_ROOM" "[researcher] getUserById returns null on missing user (auth.js:45)"

# Agent B: implementer waits for researcher, then acts
sv chat wait "$SV_CHAT_ROOM" 1          # blocks until seq 1 arrives
# ... reads findings, makes changes ...
sv chat post "$SV_CHAT_ROOM" "[implementer] Added null check + test at auth.js:45"
Tip: Chat room history persists in retained MQTT topics. Use sv chat history <room> at any time to review past decisions — even days later.

MQTT & sv Communication

All agent communication flows through the sv helper, which publishes to the Mosquitto broker. These commands are auto-approved — no supervisor delay.

Status reporting

export SV_TASK_ID="fix-auth-bug"            # set once at agent start

sv pub status started  "Investigating auth token expiry"
sv pub progress 50     "Found 3 affected files, patching now"
sv pub discovery       "Tokens expire after 1h, not 24h as documented"
sv pub status completed
Note: PostToolUse hooks auto-publish Bash/Edit/Write/Task events. You only need sv pub for high-level milestones.

Data exchange between agents

# Agent A: publish findings for Agent B
sv retain "supervisor/myproject/agent-a/findings" '{"files":["auth.js"],"issue":"..."}'

# Agent B: read them (blocks up to 30s)
sv read  "supervisor/myproject/agent-a/findings"

# Clean up
sv clear "supervisor/myproject/agent-a/findings"

Coordinator help requests

# Ask another project's agent for help
sv request "Check if getUserById throws or returns null" --project auth-service --type research
# Prints a request-id

# Wait for the response (120s timeout)
result=$(sv request wait <request-id> 120)
echo "$result"

# In the target session — respond
sv respond <request-id> "Returns null at users.js:45"

UI Panels

Operations

Unified panel combining what was previously "Coordinator" and "Agent Activity." Shows subagent tasks (dispatched, running, completed), coordinator requests (click Dispatch to start workflows), debate rounds, and live MQTT status/progress/discovery events. This is the primary view for tracking all agent work across your sessions.

Version Badge

Gold badge in the supervisor bar — shows number of pending commits since last restart. Click to see the diff. The dashboard also checks the installed Claude Code version against npm latest. An update button triggers claude update across all sessions.

Time Bar

Shown for running agents. Elapsed time vs budget. Turns amber as deadline approaches.

Eval History

Filterable log of every AI evaluation: approved, denied, or escalated, with confidence scores. Filter by project or search for specific commands.

Context % Badge

Color-coded badge on each terminal tab showing context window usage. Normal → warning (50%) → alert (65%) → critical (80%).

Rate Limits

5-hour and 7-day API usage parsed from /usage output and Anthropic OAuth headers. Shown in the dashboard header.

Team Chat

Two modes: human-to-human messaging across instances, and a live viewer of agent-to-agent chat rooms. Watch debates, collab negotiations, and coordinator exchanges unfold in real time. Toggle between Human and Agent views.

Document Viewer

Renders markdown from project docs/ directories. Can fetch docs from peer supervisor instances.

Controller / Viewer

First browser gets keyboard input; others are read-only viewers with a Take Control button.

Mobile PWA: The dashboard includes apple-mobile-web-app-capable meta tags. Add it to your home screen from Safari or Chrome to use it as a standalone app.

File and Image Drop

The terminal accepts files and images via drag-and-drop or clipboard paste. Images are saved to your project's screenshots/ directory; other files go to the drops/ directory. The file path is auto-typed into the terminal so you can append your prompt and hit Enter.

Drag and drop

Drag any file onto the terminal area. A "Drop file here" overlay appears while dragging. Release to drop — the file is saved and the path is inserted at the cursor. Images (PNG, JPG, GIF, WebP) go to screenshots/; all other files go to drops/.

Clipboard paste

Copy an image to your clipboard (e.g. take a screenshot with your OS shortcut), click the terminal to focus it, then press Ctrl+V. The image is extracted from the clipboard and handled identically to a drag-and-drop.

What happens next

The file path is typed into the terminal with a trailing space. Add your prompt text and press Enter. Claude reads images natively via the Read tool — no extra steps required.

Detail Value
Image save location screenshots/ in the project directory
Other file save location drops/ in the project directory
Supported image formats PNG, JPG, GIF, WebP
Max file size 10 MB
Tip: Take a screenshot, Ctrl+V into the terminal, then type what's wrong with this? — Claude will analyze the image.

Email Notifications

When a Claude session ends — not on every turn, only on real stops like max_turns, user interrupt, or task completion — the supervisor can send you an email with the project name, stop reason, and a summary of git state (uncommitted changes and recent commits).

How to enable

From within a session — takes effect immediately, no restart needed:

echo "you@gmail.com" > .claude/notify-email

Via environment variable — set in your systemd unit or shell environment before launching:

# In the systemd unit file:
Environment=SUPERVISOR_NOTIFY_EMAIL=you@gmail.com

# Or in your shell before launching:
export SUPERVISOR_NOTIFY_EMAIL=you@gmail.com

If both are set, .claude/notify-email takes precedence.

How to disable

# Remove the per-project file:
rm .claude/notify-email

# Or unset the environment variable:
unset SUPERVISOR_NOTIFY_EMAIL
Tip: Enable email notifications before kicking off long background work, then close your laptop. You'll get an email when it's done.

Security Policy

The AI evaluator reads supervisor-policy.md — a plain markdown file in your supervisor config directory. Edit it directly to add, remove, or modify rules. Changes take effect immediately with no restart required.

Three-tier structure

Tier Behavior Examples
Always approve Auto-approved with no delay or AI evaluation Read-only tools, sv commands, npm test, git reads
Always deny Blocked immediately, never sent to AI evaluator rm -rf on system paths, writes to .env, git push --force to main
Evaluate carefully Sent to AI evaluator — approved, denied, or escalated to human based on confidence score Piped bash commands, Docker builds, operations outside the project directory

Learned approvals

hooks/dynamic-approvals.sh is auto-generated from eval history. Commands that meet the promotion thresholds are added to the fast-path auto-approve list, bypassing the AI evaluator entirely on future calls. See the Dynamic Approvals section for details.

Tip: Add a comment to supervisor-policy.md explaining why a rule exists. The AI evaluator reads the full file for context when making close calls.

Dynamic Approvals

The supervisor learns which commands are safe from its own eval history. A housekeeping script analyzes past evaluations and promotes frequently-approved patterns to a generated fast-path file, eliminating AI evaluator overhead for routine commands.

How it works

  1. Every AI evaluation is logged to logs/eval-history.jsonl.
  2. scripts/eval-housekeeping.js runs twice daily, analyzing the log for patterns that meet the promotion thresholds.
  3. Promoted patterns are written to hooks/dynamic-approvals.sh.
  4. The pre-tool-use hook sources this file — matching commands are auto-approved without calling the AI evaluator.

Promotion thresholds

Threshold Value
Min occurrences 3
Min approval rate 90%
Min avg confidence 85%

Running manually

node scripts/eval-housekeeping.js
Tip: Run the housekeeping script manually after a period of intensive agent work to fast-track patterns you've been approving repeatedly. The generated dynamic-approvals.sh is checked into version control so learned rules persist across server restarts.

Delegation Enforcement

The supervisor enforces subagent delegation based on remaining context window percentage, so your main context stays lean.

Context thresholds

Context remaining Behavior
> 50% No enforcement — full access to all tools
30–50% Warning injected — nudges toward Task tool delegation
< 30% Implementation tools denied — corrective instructions injected automatically

Enforced patterns (denied when context is low)

  • 3 or more chained && operators in a single bash command
  • Write calls over ~150 lines
  • 3 or more sequential Write/Edit/Bash calls across multiple files without any Task tool use

NOT enforced (always allowed)

  • Single-file edits, iterative patches to one file
  • Test runs, builds, linters
  • Git operations (status, log, diff, commit)
  • API calls with piped output processing
  • Read-only work (Read, Glob, Grep)
Note: The Task tool itself is never denied — it is the delegation mechanism. Spawning a subagent whose prompt describes multi-step work is always approved.

Session Continuity

The supervisor preserves work across context compaction and session restarts through a layered recovery system.

Pre-compaction

  • Auto-commits all uncommitted work before context is compacted
  • Writes .claude/progress-snapshot.md — git state, agent activity, pending approvals
  • Extracts a decision trail from the transcript (key choices, discoveries, blockers)
  • Publishes an MQTT event so the dashboard logs the compaction

Session handoff

  • On real session end (user interrupt, max_turns, task completion — not every response turn): on-stop.sh writes .claude/session-handoff.md with stop reason, git status, uncommitted changes summary, recent commits, and a recovery checklist
  • On next session start: session-start.sh re-injects up to 3 files as system context:
    • session-handoff.md — if <48 hours old
    • progress-snapshot.md — if <24 hours old, with instruction to resume previous work
    • transcript-excerpt.md — if <24 hours old, with instruction to extract a decision trail
Key detail: The handoff only writes on real session endings, not on every response turn. This avoids overwriting a meaningful handoff with a generic "end_turn" state.

Post-compaction recovery

CLAUDE.md.template includes a recovery checklist that is re-read after every compaction. The checklist directs Claude to read the progress snapshot, check git log, verify task state, and ask the user before continuing if state is unclear.

Context tracking

The supervisor parses PTY output to extract context window percentage and saves it to .claude/context-percent.txt. Predictive warnings are injected at 50%, 65%, and 80% to prompt proactive delegation before limits are hit.

Defer / Resume

Headless Claude sessions can defer approval decisions to the supervisor instead of blocking on human input. This enables cron-triggered agents, background tasks, and CI pipelines to pause safely and resume after a decision is made.

How it works

  1. A headless session encounters a tool call requiring approval and calls POST /api/hook/defer with { approval_id, session_id }.
  2. The supervisor stores the mapping and displays the pending approval in the dashboard.
  3. A human (or the AI evaluator) resolves the approval.
  4. The server spawns claude -p --resume <sessionId> in the correct project directory.
  5. The resumed session continues exactly where it left off.

Use cases

  • Cron-triggered agents that need a human approval checkpoint before destructive operations
  • Background tasks spawned from CI pipelines that may hit policy-gated commands
  • Long-running research agents that pause for confirmation before making changes
Tip: Defer/Resume is the correct pattern for any scheduled or unattended agent work that still needs occasional human sign-off — it avoids polling loops and doesn't require the human to be present at the moment the decision is needed.

API Proxy

The supervisor includes a transparent proxy for Anthropic API calls. Set ANTHROPIC_BASE_URL to route all Claude API traffic through the supervisor so it can track per-session call counts.

Setup

export ANTHROPIC_BASE_URL=http://localhost:3847  # routes API calls through supervisor

What it does

  • Transparently proxies all /v1/* requests to https://api.anthropic.com
  • Extracts the X-Claude-Code-Session-Id header for per-session tracking
  • Tracks call counts per session — visible in /api/state under apiProxy.sessionCalls
  • Streams responses with no content modification
Detail Value
Proxy port 3847 (same port as supervisor)
Proxied path /v1/* → https://api.anthropic.com/v1/*
Session tracking header X-Claude-Code-Session-Id
State key apiProxy.sessionCalls in /api/state
Min Claude Code version v2.1.87+ (required for session ID header)
Tip: Per-session call counts let you see which agent tasks are making the most API calls. Useful for identifying runaway agents or optimizing expensive workflows.

AskUserQuestion Auto-Answer

When a worker Claude instance uses the AskUserQuestion tool, the supervisor can answer automatically without human intervention.

A secondary Claude Sonnet instance evaluates the question and injects a typed answer directly into the terminal after a configurable delay. The delay gives you a window to override the injected answer if you want to respond yourself.

Configuration

# Fully unattended mode — AI answers all questions automatically
export SUPERVISOR_AUTO_ANSWER_QUESTIONS=ai

# Human mode (default) — questions pause and wait for your input
export SUPERVISOR_AUTO_ANSWER_QUESTIONS=human
Tip: Set SUPERVISOR_AUTO_ANSWER_QUESTIONS=ai before starting long background runs. The session won't stall waiting for a human if Claude has a clarifying question mid-task.

Best Practices

Before major decisions

  • Run /feasibility <topic> before hardware purchases, API integrations, or architecture changes you can't easily undo.
  • Run /debate <topic> when you've been going back and forth — the moderator often finds middle ground neither side saw.
  • Use research coordinator type to gather facts from another project before making a cross-service change.

Keeping context lean

  • Delegate all multi-file reads, code edits, and long searches to subagents — keep your main context for coordination.
  • Commit working changes frequently. The pre-compact hook auto-commits, but don't rely on it.
  • Use /review-changes after long agent runs to see what changed before restarting.

Agent coordination

  • Use chat rooms when two or more agents need to share findings — retained messages mean nothing is lost.
  • Use sv retain/read for structured data handoffs (JSON payloads, file lists, findings).
  • Set SV_TASK_ID at the top of every subagent task — it labels all MQTT activity in the dashboard.
Note: The supervisor evaluator blocks rm -rf on broad paths, writes to .env/secrets, and git push --force to main. These require explicit human approval even in auto mode.
